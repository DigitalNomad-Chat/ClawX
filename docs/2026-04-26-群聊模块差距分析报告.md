## 三、核心差距深度分析

### 差距 G1：路由层未消费结构化输出（最关键）

**原项目流程**：

```
用户消息 → routes.ts POST /messages
  → runtimeDispatch()
    → Gateway chat.send
    → 轮询 chat.history
    → content-sanitizer 解析 <hall-structured>
    → applyStructuredBlockToTaskCard()
      → 自动更新 proposal / decision / doneWhen / nextAction
      → 根据 nextAction 自动推进状态机
```

**当前项目流程**：

```
用户消息 → routes.ts POST /messages
  → dispatchAgentRun()  ← 简化版，跳过结构化解析
    → Gateway chat.send
    → 轮询 chat.history
    → 基本清洗
    → 直接持久化消息  ← 结构化数据丢失
```

**影响**：TaskCard 的 proposal/decision/doneWhen 字段永远为空，前端决策面板无数据可展示。这是所有下游功能（决策面板、执行计划、自动推进）的数据基础。

---

### 差距 G2：SSE 流式传输是伪流式（经交叉验证确认）

**代码证据**：

| 证据位置 | 原项目 | 当前项目 |
|----------|--------|----------|
| **事件类型** | `draft_start` / `draft_delta` / `draft_complete` / `draft_abort` | `draft_chunk`（单个占位） / `draft_finalize`（整体推送） |
| **内容切分** | `chunkDraftContent()` 按 48 字符切分 + `setImmediate` 逐帧推送 | 无切分，一次性推送完整回复 |
| **runtime-dispatch.ts** | 直接流式推送 delta | 第 14 行注释："This is a non-streaming implementation"；使用 `pollHistoryForAssistantReply()` 每 3 秒轮询，最长 180 秒 |
| **stream-publisher.ts** | N/A（有独立 collaboration-stream.ts） | 文件存在但**从未被任何模块导入**，是死代码 |
| **前端体验** | Agent 回复逐字出现，打字机效果 | 显示 "正在思考..." 占位 → 等待 → 一次性显示完整回复 |

**结论**：当前项目 SSE 名义上有 `draft_chunk` / `draft_finalize` 事件，但实际只是"发一个占位文本 → 等轮询拿到完整结果 → 一次性推送"，不是真正的流式传输。

---

### 差距 G3：讨论周期未激活

**原项目实现**：

```typescript
// 1. 开启讨论 → 构建参与者队列（按角色顺序）
openDiscussionCycle(taskCard, openedBy, participants)
  → expectedParticipantIds: [planner, coder, reviewer, manager]

// 2. 意图感知调度
classifyHallDiscussionFollowupIntent(userMessage)
  → "讨论继续" → 下一位发言
  → "拍板定一下" → manager 收口
  → "直接出交付物" → 跳过讨论，直接执行
```

**当前项目现状**：
- `speaker-policy.ts` 已移植完整逻辑
- `store.ts` 有 DiscussionCycle 类型
- 但 `routes.ts` 消息派发时未检查/使用 DiscussionCycle
- 无意图分类系统

**影响**：多 Agent 无法自动按角色顺序讨论，用户 @多个 Agent 时所有 Agent 同时响应（无序）。

---

### 差距 G4：协作风格指南（HALL.md）缺失（交叉验证新发现）

**原项目实现**：

原项目根目录存在 `HALL.md`，定义了协作行为准则：
- **语气风格**：像忙碌工作聊天中的干练同事
- **讨论规则**：第二发言人应补充缺失视角而非复述
- **执行规则**：只做当前切片，产出实际交付物而非描述
- **评审规则**：先列 must-fix
- **交接规则**：说明现状、待办和下一步负责人

该文件被 `hall-runtime-dispatch.ts` 加载并注入到 system prompt 中。

**当前项目现状**：
- `prompt-builder.ts` 的 `buildSystemPrompt()` 是硬编码的角色描述
- 无加载外部 HALL.md 或类似协作风格文件的逻辑
- Agent 回复缺少风格约束，可能导致冗长、空洞、不聚焦的输出

### 差距 G6：合成打字指示器缺失（交叉验证新发现）

**原项目实现**：
- 完整 draft 生命周期：`beginHallDraftReply` → `pushHallDraftDelta` → `completeHallDraftReply`/`abortHallDraftReply`
- 区分 "real draft"（一个参与者一个）和 "placeholder/synthetic typing"（为未来发言人预显示）

**当前项目现状**：
- 仅有 routes.ts 中单个 `"正在思考…"` 占位文本推送
- 无 draft 生命周期管理
- 无多参与者占位 typing

### 差距 G7：执行链——两项目基本对等（交叉验证修正）

## 六、修复策略建议

### 核心路径

```
阶段 1 — P0 链路打通（~700 行）
  G1 路由层切换 → G2 SSE 真流式 → G3 讨论周期 → G4 执行锁 → G5 HALL.md

阶段 2 — P1 UI 增强（~900 行，依赖 P0）
  G6 三栏布局 → G7 状态条+打字 → G8 决策面板 → G9 操作按钮 → G10 Markdown → G11 执行计划编辑

阶段 3 — P2 质量保障（~600 行，可与 P1 并行）
  G12 测试覆盖 → G13 提及芯片 → G14 摘要存储
```

### 关键发现

当前项目的核心问题不是"需要从零实现"，而是：

1. **已有基础设施未被激活**（路由层未切换、stream-publisher 是死代码、讨论周期未集成）
2. **流式传输是伪实现**（占位+轮询，不是真流式）
3. **协作风格缺失**（无 HALL.md，Agent 回复缺少行为约束）
4. **测试完全空白**（零测试，任何改动都无法验证正确性）

## 七、结论

当前项目群聊模块功能覆盖度约 **45%**。

**已完成的优势项**：
- 完整的类型系统
- 运行时调度链路（prompt → Gateway → 清洗 → 解析）
- 丰富的 API 端点（20+）
- 预算治理、执行日志、任务生命周期追踪
- React 组件化前端 + Zustand 状态管理
- 国际化支持、阶段流程可视化

**核心缺失项（按影响排序）**：
1. 路由层未切换到完整版调度 → 结构化数据丢失（G1）
2. SSE 是伪流式，体验差距大（G2）
