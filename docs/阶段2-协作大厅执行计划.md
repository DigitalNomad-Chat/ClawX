# 阶段 2：协作/群聊 (Collaboration Hall) 执行计划

## 一、调研结论

### 1.1 Control Center 协作大厅全貌

Control Center 的协作大厅是一个极其复杂的子系统，核心文件及其规模：

| 文件 | 规模 | 职责 |
|------|------|------|
| `collaboration-hall-store.ts` | ~1300 行 | Hall/Message/TaskCard 的 CRUD + 校验/归一化 |
| `collaboration-hall-orchestrator.ts` | ~1400 行 | 完整编排器：讨论→指派→执行→评审流程 |
| `collaboration-stream.ts` | ~500 行 | SSE 实时流服务器（draft_start/delta/complete） |
| `hall-mention-router.ts` | ~50 行 | @提及解析（@all + 别名匹配） |
| `hall-role-resolver.ts` | ~100 行 | 从 Agent roster 解析 semantic role |
| `hall-speaker-policy.ts` | ~200 行 | 讨论轮次策略 |
| `hall-runtime-dispatch.ts` | ~400 行 | 通过 ToolClient 向 Gateway 派发 Agent 运行 |
| `hall-execution-lock.ts` | ~100 行 | 执行锁管理 |
| `hall-handoff.ts` | ~150 行 | 交接包结构化 |
| `hall-discussion-domain.ts` | ~50 行 | 讨论领域推断 |

**数据模型（3 个 JSON 文件）：**
- `collaboration-halls.json` — Hall 配置 + participants + executionLocks
- `collaboration-hall-messages.json` — 所有 HallMessage
- `collaboration-task-cards.json` — HallTaskCard

### 1.2 ClawX 现有基础设施

- **Gateway RPC**: `gatewayManager.rpc(method, params, timeout)` 通过 WebSocket
- **HTTP 代理**: `gatewayManager.httpProxy(path, init)`
- **前端 API**: `hostApiFetch` 调用 Host API
- **事件总线**: `HostEventBus`（IPC 为主，支持 `subscribeHostEvent`）
- **状态管理**: Zustand (`useChatStore`, `useAgentsStore`, `useGatewayStore`)

### 1.3 关键差异与挑战

| 差异点 | Control Center | ClawX | 适配策略 |
|--------|---------------|-------|----------|
| Gateway 通信 | ToolClient (直接调用 openclaw CLI) | GatewayManager (WS RPC + HTTP 代理) | 重写 dispatch 层，使用 `rpc()` |
| 实时流 | 自建 SSE Server (`collaboration-stream.ts`) | HostEventBus (IPC + SSE fallback) | 复用 HostEventBus，新命名空间 `collab:*` |
| Agent 列表 | agent-roster.json | `useAgentsStore` (Gateway RPC) | 从 Gateway 获取 Agent 列表生成 participants |
| 会话管理 | chat-store.ts (独立房间) | useChatStore (已集成) | 复用现有会话体系，sessionKeys 关联 |
| 存储路径 | `{runtimeDir}/collaboration-*.json` | `{userData}/modules/collaboration/*.json` | 使用 `getModuleFilePath` |

## 二、核心设计决策

### 决策 1：编排器简化策略

Control Center 的 orchestrator（1400 行）实现了完整的自动讨论-执行-评审闭环，包括自动调用 Agent、收集回复、判断阶段流转。这在 ClawX 中过于复杂且与上游 Gateway 耦合紧密。

**简化方案：**
- **保留**：讨论轮次跟踪（`discussionCycle`）、阶段状态机（`stage`）、执行项列表（`plannedExecutionItems`）
- **简化**：不实现自动 Agent 调用。用户手动点击"指派执行"后，通过 Gateway RPC 触发 `agent.run`，前端轮询或等待 Gateway 事件更新状态
- **保留**：`@handoff` 消息识别，自动推进 `currentExecutionItem` 到下一项
- **移除**：自动 draft reply、自动 speaker 选择、自动 discussion domain 推断

### 决策 2：实时流策略

不复建 SSE Server，而是：
- 后端：所有写操作通过 `HostEventBus.emit('collab:invalidate', {...})` 推送变更事件
- 前端：`subscribeHostEvent('collab:invalidate', handler)` 监听事件，触发 store 重新拉取
- 优势：零新增网络端口，复用现有 CORS/Auth 基础设施

### 决策 3：Participant 来源

- Hall 的 participants 从 ClawX 的 `listAgentsSnapshot()` 动态生成
- 映射规则：`agent.id` → `participantId`，`agent.name` → `displayName`
- Semantic role 通过名称关键词推断（`hall-role-resolver.ts` 的逻辑，50 行可直接移植）
- Human 用户固定为一个特殊 participant：`participantId = "human"`

### 子任务 2.6：Gateway 运行时调度（简化版）
**文件**：`electron/modules/collaboration/runtime-dispatch.ts`

**职责**：
- 提供 `dispatchAgentRun(participantId, taskCard, message)` 函数
- 内部调用 `gatewayManager.rpc('agent.run', {...})` 触发 Agent 执行
- **不实现**：自动轮询、自动收集回复、自动推进流程
- **仅实现**：单次触发，返回 sessionKey，由前端或用户手动跟进

**说明**：
这是与 Control Center 差异最大的部分。Control Center 的 `hall-runtime-dispatch.ts` 实现了完整的执行链自动调度（400 行）。在 ClawX 中，我们将其简化为"手动触发 + 状态跟踪"模式，降低复杂度，同时保留扩展可能性。

**工期**：0.5 天

| 风险 | 影响 | 缓解 |

| Control Center orchestrator 逻辑过于复杂，简化后功能不完整 | 中 | 明确 MVP 范围：保留数据模型和手动流程，自动调度后续迭代 |

| 与任务看板模块的数据一致性 | 中 | TaskCard 的 `projectId`/`taskId` 预留关联字段，等任务看板实现后对接 |

## 七、关键接口约定

### Gateway RPC 调用（预留适配）
```typescript
// runtime-dispatch.ts 中的调用方式
const result = await gatewayManager.rpc('agent.run', {
  agent: participant.agentId,
  message: buildPrompt(taskCard, message),
  // 其他参数根据 Gateway API 调整
}, 60000);
```

### HostEventBus 事件
```typescript
// 后端推送
eventBus.emit('collab:invalidate', {
  hallId: 'main',
  taskCardId?: string,
  messageId?: string,
  reason: 'message_created' | 'task_updated' | 'task_created' | ...
});

// 前端订阅
subscribeHostEvent('collab:invalidate', (event) => {
  dashboardStore.fetchOverview();
});
```
