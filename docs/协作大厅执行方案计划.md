# 协作大厅（Collaboration Hall）执行方案计划

> 基于《差距分析报告》制定
> 状态：待审批，审批后按阶段执行

---

## 一、总体策略

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **物理隔离** | 所有新增/修改代码严格限定在 `electron/modules/collaboration/` 和 `src/modules/collaboration/`，不动核心挂载点 |
| **渐进增强** | 每阶段完成后系统可独立运行，不依赖后续阶段 |
| **适配优先** | ClawX 没有 `ToolClient.agentRun`，Runtime Dispatch 基于现有 `gatewayManager.rpc('chat.send')` 适配实现 |
| **事件驱动** | 模块内部使用 HostEventBus 解耦，SSE 流基于事件总线构建 |
| **防御编程** | 所有自动流转支持手动覆盖，Agent 回复异常时不阻塞整个大厅 |

### 阶段 1：运行时派发引擎（Runtime Dispatch）—— 核心引擎

**目标**：实现与 Gateway 的深度集成，让 Agent 能在群聊环境中"像人一样"协作。这是整个系统最关键的部分。

**工期**：7-10 天

#### 1.1 提示词构建器（Prompt Builder）

**新增文件**：`electron/modules/collaboration/prompt-builder.ts`

职责：根据 mode（discussion / execution / handoff / review）构建差异化 prompt。

```typescript
export type DispatchMode = 'discussion' | 'execution' | 'handoff' | 'review';

export interface PromptBuildInput {
  mode: DispatchMode;
  participant: HallParticipant;
  taskCard: HallTaskCard;
  hall: CollaborationHall;
  triggerMessage?: HallMessage;
  recentMessages?: HallMessage[]; // 最近 N 条消息作为上下文
  currentExecutionItem?: HallExecutionItem;
  operatorIntent?: HallOperatorIntent; // 用户直接指令意图

  systemPrompt: string;
  userMessage: string;
  mode: DispatchMode;
  expectedOutputFormat: 'structured' | 'free';
}

export function buildDispatchPrompt(input: PromptBuildInput): BuiltPrompt;
```

**实现要点**：
- discussion 模式：鼓励多角度分析，不要求具体交付物，prompt 引导 Agent 发表观点
- execution 模式：明确要求具体交付物（代码/文案/URL），prompt 包含 `doneWhen` 和当前任务
- handoff 模式：引导 Agent 总结当前进度，明确下一步交接对象
- review 模式：引导 Agent 评审上一个执行者的产出，指出 must-fix 项
- 所有模式 prompt 末尾要求输出 `<hall-structured>` JSON 块

#### 1.6 重写 Runtime Dispatch

**重写文件**：`electron/modules/collaboration/runtime-dispatch.ts`

职责：整合以上所有模块，实现完整的 Agent 派发流程。

```typescript
export interface RuntimeDispatchInput {
  ctx: HostApiContext;
  mode: DispatchMode;
  participant: HallParticipant;
  taskCard: HallTaskCard;
  hall: CollaborationHall;
  triggerMessage?: HallMessage;
  dispatch?: boolean; // 是否实际调用 Gateway（false 仅构建 prompt 用于预览）
}

export interface RuntimeDispatchResult {
  success: boolean;
  message?: HallMessage;      // 生成的消息（终稿）
  sessionKey: string;
  structured?: ParsedStructuredBlock;
  nextAction?: 'continue' | 'review' | 'blocked' | 'handoff' | 'done';
  error?: string;
}

export async function runtimeDispatch(input: RuntimeDispatchInput): Promise<RuntimeDispatchResult>;
```

**完整流程**：

```
1. 检查 Gateway 状态
2. 解析/生成 sessionKey（agent:${agentId}:hall:${taskId}）
3. 获取最近消息历史（从 messageStore 读取）
4. 构建差异化 prompt（prompt-builder）
5. 调用 gatewayManager.rpc('chat.send', { sessionKey, message, deliver: true })
6. 【流式】实时推送草稿到 SSE（stream-publisher）
7. 接收完整回复
8. 解析 <hall-structured> 块（structured-parser）
9. 清洗内容（content-sanitizer）
10. 交付物强制检测（deliverable-enforcer）
11. 生成 HallMessage 并持久化
12. 返回结果（含 nextAction 供编排器使用）
```

**关键适配点**：
- ClawX 的 `chat.send` 不是流式返回，需要轮询 session history 获取增量内容来模拟流式效果
- 或者：先实现非流式版本（一次性返回），后续再优化为流式

**建议**：先实现非流式版本（降低复杂度），阶段 3 再叠加流式能力。

#### 2.2 编排调度器

**新增文件**：`electron/modules/collaboration/orchestrator-scheduler.ts`

职责：定时检查所有 TaskCard 的编排状态，处理超时和自动推进。

```typescript
export interface SchedulerOptions {
  checkIntervalMs: number;    // 默认 30s
  discussionTimeoutMs: number; // 默认 5min
  executionTimeoutMs: number;  // 默认 10min
  reviewTimeoutMs: number;     // 默认 5min
}

export function startOrchestratorScheduler(ctx: HostApiContext, opts?: SchedulerOptions): () => void;
export function stopOrchestratorScheduler(): void;
```

**调度逻辑**：
- 每 30 秒扫描一次所有非终态 TaskCard
- 检查每个 TaskCard 的 `enteredAt` 是否超过对应阶段的 timeout
- 超时后触发 `timeout` 事件，编排器决定下一步（通常是推进或告警）
- discussion 阶段：检查是否已收集足够的回复数（默认 2 个非人类 Agent 的回复）
- execution 阶段：检查执行锁是否超时，超时自动释放并标记 blocked

## 四、关键接口设计

### 4.1 新增 REST API

```
POST /api/collaboration/task-cards/:id/auto-assign
  Body: { dispatch?: boolean }
  Response: { success: true, taskCard: HallTaskCard, assignedParticipant?: HallParticipant }

POST /api/collaboration/task-cards/:id/auto-advance
  Body: { }
  Response: { success: true, taskCard: HallTaskCard, newState: string }

POST /api/collaboration/task-cards/:id/dispatch
  Body: { participantId: string, mode?: 'discussion'|'execution'|'handoff'|'review', message?: string }
  Response: { success: true, result: RuntimeDispatchResult }

GET /api/collaboration/stream?hallId=main
  Response: text/event-stream
  Events:
    - draft_chunk { draftId, chunk, participantId, taskCardId }
    - draft_finalize { draftId, message, participantId, taskCardId }
    - message_created { message }
    - task_updated { taskCard }
    - orchestrator_state_change { taskCardId, state, previousState }

GET /api/collaboration/orchestrator/status
  Response: { success: true, states: Array<{ taskCardId, state, enteredAt, timeoutAt }> }
```

### 4.2 内部事件（HostEventBus）

```
collab:draft:chunk      → StreamManager 订阅 → SSE 推送
collab:draft:finalize   → StreamManager 订阅 → SSE 推送
collab:draft:abort      → StreamManager 订阅 → SSE 推送
collab:message:created  → 已存在，保持不变
collab:task:updated     → 已存在，保持不变
collab:orchestrator:state → StreamManager 订阅 → SSE 推送
```

## 八、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| `chat.send` 不支持流式，SSE 草稿效果受限 | 中 | 阶段 1 先实现非流式，阶段 3 通过轮询 session history 模拟流式 |
| 编排器自动流转逻辑复杂，易引入死循环 | 高 | 每个状态设置超时 + 最大重试次数（默认 2 次）+ 手动 override 兜底 |
| Gateway RPC 失败导致任务卡住 | 中 | Runtime Dispatch 捕获所有异常，失败时标记 blocked 并通知 |
| 交付物检测误报/漏报 | 中 | 保留手动"强制通过"按钮，用户可覆盖自动检测 |
| 上游 ClawX 更新 | 低 | 所有代码在独立目录，不修改核心文件 |
| 多 Agent 并发执行导致数据竞争 | 中 | ExecutionLock 机制（阶段 0），同一 TaskCard 同时只允许一个执行者 |

## 九、执行建议

### 建议 1：分阶段交付，每阶段独立可验证

不要等所有阶段完成再测试。每阶段完成后立即验证，确保基础稳固。

### 建议 2：阶段 1 采用"非流式先行"

Runtime Dispatch 第一阶段先实现非流式（一次性返回），降低复杂度。等非流式稳定后再叠加流式能力。

### 建议 3：编排器采用"手动为主、自动为辅"

初期编排器以手动触发为主（前端提供 assign/handoff/review 按钮），自动检测作为辅助（超时提醒、讨论完成提示）。等稳定后再提高自动化程度。

### 建议 4：交付物检测先覆盖常见类型

不必一次性实现参考实现的所有交付物类型。先覆盖 `generic`（列表/URL/代码块）和 `review`（must-fix 关键词），再逐步增加 spoken_openings、thumbnail 等垂直类型。
