## 一、核心发现与策略

### 1.1 关键发现

经过对当前代码的逐文件审计，核心发现如下：

| 发现 | 说明 |
|------|------|
| **类型已就绪** | `types.ts` 已包含 `ExecutionLock`、`StructuredHandoffPacket`、`TaskDiscussionCycle`、`HallExecutionItem` 等完整类型定义 |
| **核心逻辑已存在** | `runtimeDispatch` 完整链路已实现：prompt 构建 → Gateway 调用 → 历史轮询 → 内容清洗 → 结构化解析 → TaskCard 更新 |
| **链路未打通** | `routes.ts` 的消息派发入口仍在使用简化的 `dispatchAgentRun`，绕过了结构化解析 |
| **前端未消费** | 结构化字段（proposal/decision/doneWhen）已在后端生成，但前端仅展示纯文本 |
| **编排器已就绪** | `orchestrator.ts` 包含完整状态机和执行锁逻辑，但未被路由层主动调用 |

**结论**：大量优化是"打通现有链路"而非"从零实现"，预计工作量远小于初始评估。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **打通优先** | 优先将已存在但未激活的链路接通，而非编写新代码 |
| **物理隔离** | 所有修改限定在 `electron/modules/collaboration/` 和 `src/modules/collaboration/` |
| **渐进增强** | 每阶段完成后系统可独立运行，不依赖后续阶段 |
| **防御编程** | 所有自动流转支持手动覆盖，Agent 回复异常时不阻塞整个大厅 |
| **适配优先** | 基于现有 `gatewayManager.rpc('chat.send')` + `pollHistoryForAssistantReply` 适配 |

## 三、分阶段执行计划

### 阶段 P0：核心链路打通（预计 1-2 周）

> **目标**：让已有的后端结构化能力真正运行起来，打通"Agent 回复 → 结构化解析 → TaskCard 自动更新"完整链路。

#### 任务 P0.1：路由层切换 dispatchAgentRun → runtimeDispatch

**修改文件**：`electron/modules/collaboration/routes.ts`

**改动点**：

```
routes.ts → POST /api/collaboration/messages
  当前: dispatchAgentRun({ ctx, participant, taskCard, message })
  改为: runtimeDispatch({
    ctx, mode: "discussion", participant, taskCard, hall,
    operatorIntent: { type: "direct_ask", text: message },
    triggerMessage: savedMessage
  })
```

**具体步骤**：
1. 在 `POST /messages` 的 mention 派发分支中，构建 `CollaborationHall` 对象（从 store 读取）
2. 调用 `runtimeDispatch` 替代 `dispatchAgentRun`
3. 从 `RuntimeDispatchResult` 中提取 `structured` 字段，通过 SSE 推送给前端
4. `dispatchAgentRun` 标记为 `@deprecated`，不删除（向后兼容）

**验收标准**：
- [ ] @Agent 发送消息后，TaskCard 的 proposal/decision/doneWhen 字段自动更新
- [ ] 结构化字段通过 SSE `draft:finalize` 事件推送到前端
- [ ] 旧版 `dispatchAgentRun` 仍可通过直接调用使用

#### 任务 P0.2：集成 DiscussionCycle 追踪

**修改文件**：
- `electron/modules/collaboration/routes.ts`
- `electron/modules/collaboration/orchestrator.ts`
- `electron/modules/collaboration/store.ts`

**改动点**：

1. `store.ts` — 新增 `openDiscussionCycle` / `closeDiscussionCycle` 函数：
   ```ts
   async function openDiscussionCycle(taskCardId: string, expectedParticipantIds: string[]): Promise<TaskDiscussionCycle>
   async function closeDiscussionCycle(taskCardId: string): Promise<void>
   async function recordDiscussionResponse(taskCardId: string, participantId: string): Promise<void>
   ```

2. `routes.ts` — 在 `POST /task-cards/:id/open-discussion` 中调用 `openDiscussionCycle`

3. `orchestrator.ts` — `autoAdvanceTask` 检查 `discussionCycle.completedParticipantIds` 是否覆盖 `expectedParticipantIds`，全覆盖时自动关闭

4. `routes.ts` — `POST /messages` 派发前检查 `discussionCycle`，确定当前轮到谁发言

**验收标准**：
- [ ] 开启讨论后，只有当前轮次参与者会被派发
- [ ] 所有预期参与者回复后，讨论周期自动关闭
- [ ] 超时（可配置，默认 10 分钟）后周期自动关闭并通知

#### 任务 P0.3：统一执行锁使用

**修改文件**：
- `electron/modules/collaboration/routes.ts`
- `electron/modules/collaboration/orchestrator.ts`

**改动点**：

1. `routes.ts` — 所有派发入口（dispatch、handoff、execution）加锁检查：
   ```ts
   const activeLock = getActiveLock(taskCard.taskCardId);
   if (activeLock && activeLock.participantId !== participant.participantId) {
     return res.status(409).json({ error: "任务已被其他 Agent 锁定" });
   }
   ```

2. `routes.ts` — `POST /task-cards/:id/dispatch` 自动获取锁：
   ```ts
   await acquireExecutionLock(taskCard.taskCardId, participant, "execution");
   ```

3. `routes.ts` — `POST /task-cards/:id/handoff` 自动释放锁：
   ```ts
   await releaseExecutionLock(taskCard.taskCardId, "handoff_complete");
   ```

4. `orchestrator.ts` — `dispatchToParticipant` 中集成锁获取/释放

**验收标准**：
- [ ] 同一 TaskCard 同时只有一个 Agent 持有执行锁
- [ ] 锁过期（默认 30 分钟）后自动释放
- [ ] 手动释放锁的 API 可用

**改动点**：

## 六、风险与回滚

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| `runtimeDispatch` 性能不如 `dispatchAgentRun` | 低 | 中 | 保留旧函数作为 fallback，通过配置开关切换 |
| DiscussionCycle 导致 Agent 等待过久 | 中 | 低 | 设置超时（10 分钟），超时后自动推进 |
| 执行锁死锁 | 低 | 高 | 锁自动过期（30 分钟），提供手动释放 API |
| Markdown 渲染 XSS | 低 | 高 | 使用 `react-markdown` 的安全模式，禁止 HTML |
| SSE 事件风暴 | 低 | 中 | 事件节流，前端 debounce |

**回滚策略**：
- 每个阶段完成后创建 Git tag（`collab-p0`、`collab-p1` 等）
- 任何阶段出问题可回滚到上一个 tag
- P0 阶段保留 `dispatchAgentRun` 作为降级路径
