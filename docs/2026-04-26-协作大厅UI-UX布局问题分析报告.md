# 协作大厅 UI/UX 布局问题分析报告

> 分析时间: 2026-04-26
> 分析范围: `src/modules/collaboration/` 全部前端组件
> 分析方法: 基于前端设计原则（排版、色彩、动效、空间构图、背景细节）逐一审查

---

## 一、整体布局架构问题

### 1.1 页面层级嵌套过深，有效视口被压缩

**现状：**
```
ModulePageLayout（title + padding-6）
  └── h-[calc(100%-2rem)]（强制高度计算）
       ├── 左栏 MessageStream（自有 header + border + padding）
       │    └── 内部再嵌套 header + messages + input
       └── 右栏 任务面板（自有 header + border + padding）
```

**问题：**
- `ModulePageLayout` 自带 `p-6`（24px）内边距，加上各子组件各自的 `border-b`、`px-4 py-3`，导致页面实际可用空间被层层蚕食
- `h-[calc(100%-2rem)]` 是一个硬编码的"补丁"，说明高度计算有问题，而且 `2rem` 的减值是拍脑袋的，不同屏幕尺寸下表现不一致
- 左右两栏都使用 `rounded-lg border bg-card`，在视觉上形成两个"盒子"，看起来像两个独立模块而非一个统一工作空间

**建议：**
- 移除 `ModulePageLayout` 的 `p-6`，改用 `p-0` 或极小内边距，让工作区占满视口
- 左右两栏共享一个无边框的连续空间，仅用分割线区分
- 使用 `h-full` 配合 flexbox 自然分配高度，而非 `calc()` 硬编码

### 1.2 左右分栏比例僵化

**现状：** 左栏 `flex-1` + 右栏 `w-80`（320px 固定）

**问题：**
- 右栏 320px 在大屏上过窄（任务卡内容挤在一起），在小屏上占比过大
- 任务卡描述使用 `line-clamp-2`，320px 宽度下只能显示约 40 个汉字就截断
- 没有响应式断点，移动端/小屏完全不可用
- 左栏消息气泡 `max-w-[80%]`，在宽屏上浪费右侧空间

**建议：**
- 右栏改为 `w-80 lg:w-96`，或使用 `min-w-72 max-w-96` 的弹性宽度
- 添加 `@media` 断点：小屏时右栏折叠为底部弹出抽屉
- 考虑可拖拽的分隔线，让用户自定义比例

## 二、组件级问题

### 2.1 MemberStatusBar — 信息密度过高，缺乏层级

**现状：**
```tsx
// 横向滚动的圆角按钮组
<div className="flex items-center gap-2 overflow-x-auto px-1 py-1.5 text-[11px]">
```

**问题：**
- 所有成员状态以相同权重平铺，无法快速识别"谁正在活跃"
- `text-[11px]` + `h-4 w-4` 头像在视觉上太小，信息可读性差
- `overflow-x-auto` 意味着成员多时需要横向滚动，但没有任何视觉提示告诉用户"还有更多"
- 状态图标（`Loader2`/`CheckCircle2`/`WifiOff`）尺寸仅 `h-2.5 w-2.5`（10px），几乎看不清
- 点击行为不直观（点击清除状态），缺乏 hover 提示

**建议：**
- 活跃成员（typing/executing）使用更大的视觉权重（发光边框、脉冲动画）
- 添加"收起/展开"功能，默认只显示活跃成员
- 使用 tooltip 展示完整状态信息而非依赖 title 属性
- 考虑将"在线成员"和"活跃操作"分层展示

### 2.2 MessageStream — 消息布局的单调性

**现状：**
- 所有消息使用统一的气泡样式（`bg-muted` 圆角矩形）
- Agent 消息和用户消息仅通过 `flex-row-reverse` 区分方向
- 头像统一为 `h-8 w-8` 的像素风格圆

**问题：**
- **缺乏角色区分度**：planner/coder/reviewer/manager 的消息视觉上几乎无法区分，仅靠一个 10px 的文字标签（`RoleBadge`）
- **信息流密度低**：每条消息间距 `py-3`，加上头像 + 标签行 + 气泡，大量空间浪费在装饰性元素上
- **Markdown 渲染样式粗糙**：`prose prose-sm` 是 Tailwind 默认排版，代码块、列表、标题样式与整体设计语言脱节
- **Draft 消息与正式消息视觉无差异**：`DraftMessageItem` 只是多了一个闪烁光标，用户无法区分"正在生成"和"已确认"
- **系统消息缺乏层次**：系统消息（如"已指派 Agent X"）使用居中灰色小字，在长对话中容易被忽略

**建议：**
- 为不同角色使用不同的气泡颜色/左边框色条，形成视觉通道
- 减少 `py-3` 间距，连续同一作者的消息合并（像 Slack 那样省略重复头像）
- Draft 消息使用半透明背景 + 呼吸动画，与正式消息形成明确区分
- 系统消息使用不同宽度（如 `max-w-[60%]`）和图标背景色条

### 2.3 TaskCardItem — 信息架构扁平化

**现状：**
```tsx
// 每个任务卡 ~100px 高度
<div className="group cursor-pointer rounded-lg border p-3">
  <StageBadge />
  <CompactStageFlow />  // 4个小圆点
  <h4>标题</h4>
  <p>描述（line-clamp-2）</p>
  <div>负责人 | 状态</div>
</div>
```

**问题：**
- **视觉优先级混乱**：StageBadge 和 CompactStageFlow 功能重叠（都表示阶段），同时出现浪费空间
- **CompactStageFlow 几乎无用**：4 个 1.5px 的小圆点（`h-1.5 w-1.5`）太小，用户无法直观理解进度
- **Eye 按钮（查看详情）依赖 hover**：`opacity-0 group-hover:opacity-100`，触屏和键盘用户无法发现
- **描述文字被截断后无展开方式**：`line-clamp-2` 后无法查看完整描述

**建议：**
- 移除 CompactStageFlow，将阶段信息整合到 StageBadge 中（加进度指示）
- Eye 按钮改为始终可见的半透明图标，或使用点击任务卡直接打开详情的模式
- 添加点击展开/收起功能，替代 line-clamp-2 的粗暴截断

### 2.4 TaskCardDetail — 信息过载的"超级抽屉"

**现状：**
- Sheet 抽屉包含 ~10 个 section：阶段流程、元信息、决策面板、预算、回滚计划、阻塞项、摘要、执行计划、讨论周期、执行历史、产物、操作区
- 所有 section 线性堆叠，无 tab/折叠分组
- 操作区需要先选择参与者（下拉框），再点击具体操作按钮

**问题：**
- **认知负荷极高**：用户打开一个抽屉看到 10+ 个 section，不知道从哪里开始
- **操作入口过深**：核心操作（指派、通过、驳回）被埋在页面底部，需要滚动才能看到
- **参与者选择器不直观**：`<select>` 下拉框与操作按钮分离，用户需要先选择人、再看哪些按钮可用，心理模型不自然
- **Separator 滥用**：3 个 `<Separator />` 将内容机械分割，缺乏语义化的分组

**建议：**
- 使用 Tab 分组：`概览` / `执行计划` / `历史` / `产物`，而非线性堆叠
- 将核心操作按钮固定在抽屉底部（sticky footer），不随滚动消失
- 参与者选择改为点击头像/卡片方式，而非下拉框
- 非活跃 section（如预算、回滚计划）默认折叠

### 2.5 ExecutionPlanView — 编辑模式体验割裂

**现状：**
- 查看模式：可展开的步骤卡片
- 编辑模式：带拖拽手柄 + 上下箭头的编辑行

**问题：**
- **状态切换太突兀**：从查看模式进入编辑模式，整个 UI 完全重构，用户会感到"换了一个页面"
- **拖拽手柄是假的**：`GripVertical` 图标仅作装饰，实际只能用上下箭头移动，暗示与实际不符
- **编辑步骤时外层卡片变大**：textarea + select + button 组合让单行高度翻倍，破坏列表节奏
- **添加步骤按钮过于朴素**：`variant="ghost"` 在列表底部容易被忽略

**建议：**
- 考虑 inline editing：点击某个字段直接进入编辑，而非切换全局编辑模式
- 移除 GripVertical 装饰性图标，或引入真正的拖拽排序（`@dnd-kit`）
- 添加步骤使用虚线边框卡片 + 加号图标的明显入口

### 2.6 TutorialPanel — 导航体验问题

**现状：**
- 左侧 144px 导航 + 右侧内容区
- 导航项使用文字 + 小图标

**问题：**
- **导航和内容比例失衡**：在 `sm:max-w-lg`（512px）的 Sheet 中，左侧导航占 144px（28%），内容区只剩 ~350px
- **内容区缺乏标题**：切换 section 后内容区没有大标题，用户不知道自己在看哪个章节
- **没有进度指示**：6 个章节看完几个、还剩几个，用户无法感知
- **移动端体验差**：小屏幕上 144px 导航 + 内容区的双栏布局无法使用

**建议：**
- 改为顶部 Tab 式导航（水平排列 6 个标签），释放内容区宽度
- 或使用步骤式（上一步/下一步），更适合教程场景
- 添加章节标题和"第 X/6 章"的进度指示

### 2.7 DecisionPanel — 视觉权重不足

**现状：**
- 可折叠面板，折叠时显示一个小圆点 + 状态文字
- 展开后使用 `bg-blue-50/50`、`bg-emerald-50/50`、`bg-amber-50/50` 的浅色背景

**问题：**
- **折叠状态的视觉存在感几乎为零**：一个 1.5px 的圆点 + 10px 文字，在消息流中被淹没
- **header 行有两个重复的状态圆点**（`bg-current` + `config.dotColor`），代码中有 bug
- **在 MessageStream 中 DecisionPanel 嵌入位置不合理**：位于消息 header 和消息列表之间，打断了阅读流

**建议：**
- 折叠状态使用更明显的徽章样式（带背景色和边框）
- 修复 header 行的重复圆点 bug
- 在 MessageStream 中将 DecisionPanel 移到右侧任务面板中（或使用浮动卡片），不打断消息流

## 三、视觉设计问题

### 3.1 排版（Typography）

| 问题 | 影响 |
|------|------|
| 全页面使用系统默认字体（`text-sm` = 14px），无任何字体层次 | 标题、正文、注释视觉上无区分 |
| `text-[10px]`、`text-[11px]` 大量出现 | 过小的文字在 Retina 屏幕上勉强可读，非 Retina 上几乎看不清 |
| `text-[9px]` 在 StatusBadge 中使用 | 低于可读性下限（12px） |
| 所有标题都是 `font-medium`，没有 `font-bold` 或 `font-light` 的层次变化 | 信息权重扁平化 |

### 3.2 色彩（Color）

| 问题 | 影响 |
|------|------|
| 五种角色色系（蓝/绿/琥珀/紫/灰）+ 五种阶段色系（蓝/绿/琥珀/红/灰）几乎重叠 | 用户无法区分"角色颜色"和"阶段颜色" |
| `bg-muted` 同时用于：系统消息气泡、空状态背景、输入框背景 | 缺乏语义区分 |
| primary 色的使用过于分散（按钮、badge、活跃状态、选中态），导致"一切都在强调" = "没有重点" | 视觉焦点缺失 |

### 3.3 动效（Motion）

| 问题 | 影响 |
|------|------|
| 唯一的动画效果是 `animate-spin`（加载指示器）和 `animate-pulse`（光标闪烁） | 页面感觉"死板"，缺乏活力 |
| Sheet/抽屉的打开/关闭无自定义动画 | 使用 shadcn 默认的 slide-from-right，与其他 UI 元素无协调 |
| 状态变化（stage 从 discussion → execution）无过渡动画 | 用户可能错过关键状态变更 |
| MemberStatusBar 的状态更新无任何动画 | 状态变化无法引起注意 |

### 3.4 空间构图（Spatial Composition）

| 问题 | 影响 |
|------|------|
| 所有元素都是矩形 + 圆角（`rounded-lg`/`rounded-md`），无任何非矩形构图 | 页面感觉像"填表格" |
| 左右两栏完全对称的 `border` 包围，像两个独立窗口 | 缺乏"工作空间"的整体感 |
| 没有使用重叠、阴影、层级来创建深度 | 所有内容在同一 Z 平面上 |
| padding 值在组件间不统一：`p-2`/`p-2.5`/`p-3`/`p-4`/`px-4 py-3` | 间距节奏混乱 |

### 3.5 背景与视觉细节

| 问题 | 影响 |
|------|------|
| 纯白/纯色背景，无任何纹理或层次 | 页面平坦、缺乏氛围 |
| `bg-card` 和 `bg-background` 混用但不统一 | 暗色模式下可能产生意外的色差 |
| 没有使用渐变、毛玻璃、噪声纹理等手段创建视觉层次 | 设计语言停留在"功能原型"阶段 |

## 四、交互设计问题

### 4.1 操作发现性问题

| 操作 | 入口位置 | 发现难度 |
|------|----------|---------|
| 查看任务详情 | 点击任务卡的 Eye 按钮（仅 hover 显示） | ⚠️ 高 |
| 创建新任务 | 点击右栏 header 的 "+" 按钮（无文字标签） | ⚠️ 中 |
| 指派 Agent | 打开详情抽屉 → 选择参与者 → 点击"指派" | ⚠️ 极高 |
| 编排执行顺序 | 打开详情抽屉 → 点击"编排执行顺序"按钮 | ⚠️ 极高 |
| 查看使用教程 | 点击右栏 header 的 "?" 按钮（无文字标签） | ⚠️ 中 |
| 切换语言 | 点击右栏 header 的 "中"/"EN" 文字按钮 | ⚠️ 低 |
| @提及 Agent | 在输入框输入 `@` 弹出选择器 | ✅ 低 |

### 4.2 反馈缺失

- **操作成功/失败无 Toast 通知**：创建任务、指派、审核等操作无任何视觉反馈
- **SSE 断线无明显提示**：只有一个小 `WifiOff` 图标在消息 header 中，容易被忽略
- **加载状态不统一**：有的地方用 `animate-spin`，有的地方用 `disabled`，有的地方无反馈

### 4.3 键盘可访问性

- Tab 键在消息流中的导航不可用（消息项无 `tabIndex`）
- 任务卡详情中的 `<select>` 无法通过键盘快速选择参与者
- MentionInput 的弹出选择器虽然支持上下键，但 focus 管理不完善

## 五、总结与优先级建议

### 严重程度分级

| 级别 | 问题 | 影响面 |
|------|------|--------|
| **P0 — 阻塞体验** | TaskCardDetail 操作入口过深（4.1） | 核心功能不可用 |
| **P0 — 阻塞体验** | 右栏固定 320px 导致内容截断（1.2） | 信息不可读 |
| **P1 — 显著降级** | 页面有效视口被压缩（1.1） | 全局空间浪费 |
| **P1 — 显著降级** | MemberStatusBar 信息密度低（2.1） | 状态感知差 |
| **P1 — 显著降级** | 消息缺乏角色区分度（2.2） | 对话混乱 |
| **P1 — 显著降级** | 操作无 Toast 反馈（4.2） | 操作不确定感 |
| **P2 — 体验优化** | DecisionPanel header bug（2.7） | 视觉瑕疵 |
| **P2 — 体验优化** | ExecutionPlanView 编辑体验（2.5） | 编辑效率低 |
| **P2 — 体验优化** | TutorialPanel 布局（2.6） | 教程可用性 |
| **P3 — 精细化** | 排版层次、色彩区分、动效、空间构图（3.x） | 视觉品质 |

### 推荐改造方向

1. **去边框化**：移除左右两栏的独立边框，改为统一的沉浸式工作空间
2. **操作前置**：将核心操作按钮固定在可视区域，减少点击深度
3. **角色可视化**：为不同 Agent 角色建立明确的视觉语言（色条、图标、布局差异）
4. **动效注入**：为状态变更、消息到达、阶段切换添加微动效
5. **响应式**：至少支持 768px/1024px/1440px 三个断点
