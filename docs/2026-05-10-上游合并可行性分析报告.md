## 三、冲突详细分析

### 3.1 冲突严重程度分布

| 级别 | 数量 | 说明 |
|------|------|------|
| **CRITICAL** | 1 | 双方对同一架构做了不同方向的重构 |
| **HIGH** | ~10 | 双方修改了同一函数/组件的核心逻辑 |
| **MEDIUM** | ~20 | 逻辑有重叠但不严重 |
| **LOW** | ~120+ | 纯文本/文档/JSON，容易手动解决 |

---

### 3.2 CRITICAL 级 — 需高度关注

#### `electron/utils/openclaw-auth.ts`

| 方面 | upstream | 我们 |
|------|----------|------|
| 改动量 | +844/-154 行 | +284/-68 行 |
| 内容 | 完整重写 plugin 发现系统、MiniMax OAuth、pi-ai model cost | 较简化的改动 |

**风险**: 变更量最大的冲突文件，双方对 plugin 发现逻辑做了不同方向的重构。

## 四、有利因素

1. **83 个文件完全一致** — 通过 `sync(main)` 同步的代码，不会产生实际冲突
2. **我们的模块系统是增量式的** — `initModules`、`registerModuleIpcHandlers`、`moduleRoutes`、`moduleNavItems` 等设计为"追加模式"，可以叠加到 upstream 上
3. **upstream 的架构更成熟** — 文件预览、skill 选择器、模型选择器等都是独立功能模块，不会与我们冲突
4. **i18n JSON 冲突无风险** — 纯 key-value，机械合并
5. **NiceAI 主题与 upstream 设计 token 不冲突** — 两套 token 可共存

## 五、推荐合并策略

### 方案: Rebase onto upstream/main

```
git checkout feature/independent-kernel
git checkout -b feature/independent-kernel-backup   # 备份
git checkout feature/independent-kernel
git rebase upstream/main
```

将我们的 24 个 commit 逐个重放到 upstream/main 上，按顺序解决冲突。

### 冲突解决优先级

1. 83 个一致文件 → 自动解决
2. LOW 级文档/JSON → 机械合并（~30 分钟）
3. MEDIUM 级配置 → 手动合并（~1-2 小时）
4. HIGH/CRITICAL 级核心文件 → 仔细审查（~3-4 小时）

### 各 HIGH/CRITICAL 文件的处理原则

| 文件 | 处理方式 |
|------|---------|
| `electron/utils/openclaw-auth.ts` | 以 upstream 为基底，审查我们独有改动 |
| `electron/main/ipc-handlers.ts` | 以 upstream 为基底，补入 registerModuleIpcHandlers |
| `electron/gateway/manager.ts` | 以 upstream 为基底（含 capability monitor） |
| `electron/gateway/config-sync.ts` | 以 upstream 为基底，审查方法签名差异 |
| `src/pages/Chat/index.tsx` | 以 upstream 为基底（功能更丰富） |
| `src/pages/Chat/ChatInput.tsx` | 以 upstream 为基底（skill/模型选择器） |
| `src/stores/chat.ts` | 以 upstream 为基底（baseline-cache 等高级特性） |
| `src/components/layout/Sidebar.tsx` | 以 upstream 为基底，补入 moduleNavItems + 语义化 HTML |
| `src/App.tsx` | 合并双方路由 |
| `package.json` | 以 upstream 版本号和依赖为基底，补入我们独有依赖和脚本 |

## 八、结论

**可行，需要投入约 1 个工作日。**

核心风险集中在 `openclaw-auth.ts`（plugin 发现系统重写）和 Chat 页面（双方都大幅改动）。建议在非紧急迭代窗口执行，合并前创建完整备份分支 `feature/independent-kernel-backup`。
