# ClawX 独立内核模式 — 项目可行性分析报告

> 编制日期：2026-04-20
> 状态：已确认，选定方案 A-1（Sidecar + Node.js 内核）

---

## 一、现状分析

### 1.1 ClawX 当前架构

| 层级 | 技术栈 | 说明 |
|------|--------|------|
| **前端** | React 19 + Vite + TypeScript + Tailwind | 页面: Chat, Agents, Channels, Skills, Cron, Settings, Models |
| **桌面** | Electron 40 | Main/Renderer/Preload 三进程架构 |
| **后端通信** | IPC + WebSocket | 通过 `host-api.ts` / `api-client.ts` 统一访问 |
| **AI 运行时** | OpenClaw Gateway (Node.js, 端口 18789) | 外部依赖，ClawX 只做 GUI 管理 |
| **存储** | electron-store (JSON) + OS Keychain | 无数据库 |
| **Agent 体系** | 文件驱动 (IDENTITY.md, SOUL.md, TOOLS.md, AGENTS.md 等) | 配置对用户完全可见 |

**关键限制**：ClawX 是一个纯 GUI 管理工具，AI 推理完全依赖外部 OpenClaw Gateway。它自身没有 Agent 执行引擎。

### 1.2 参考项目架构对比

| 维度 | ClawX (当前) | ChatClaw (参考) | Clawlite (参考) | OpenHarness (参考) |
|------|-------------|-----------------|-----------------|-------------------|
| **技术栈** | Electron + React | Wails + Go + Vue | Tauri + Rust + Go | Python + Textual |
| **自建内核** | 无（纯GUI） | 有 (cloudwego/eino) | 有 (Go Agent 进程) | 有 (Python engine) |
| **AI调用** | 依赖 Gateway | 内置多模型 | 内置 Anthropic/OpenAI | 内置 Anthropic/OpenAI |
| **Agent 执行** | Gateway 管理 | 内置 Engine | Go ReAct 循环 | Python ReAct 循环 |
| **进程隔离** | 单进程 | Go 进程内 | WS 独立进程 | 进程内 |
| **Agent 封装** | 文件可读 | 编译二进制 | 代码混淆 | 源码可读 |

### 1.3 OpenClaw Agent 体系分析

当前 Agent 以文件系统方式组织，每个 Agent 目录包含：

| 文件 | 用途 |
|------|------|
| `IDENTITY.md` | 角色身份（名称、性格、Emoji） |
| `SOUL.md` | 系统提示词核心（身份、使命、原则、约束） |
| `TOOLS.md` | 可用工具与素材库 |
| `AGENTS.md` | 工作手册（会话规则、决策树、质量门控） |
| `USER.md` | 用户画像与偏好 |
| `HEARTBEAT.md` | 心跳/定时任务 |
| `MEMORY.md` | 长期记忆与经验积累 |

这些文件完全明文暴露在 `~/.openclaw/workspace/agents/` 目录下。

## 二、核心需求

| # | 需求 | 关键约束 |
|---|------|---------|
| N1 | **独立内核** — 内置 Agent 执行引擎，不依赖外部 Gateway | 不能大幅修改原 ClawX 核心代码 |
| N2 | **预置 Agent** — 开箱即用的角色化 Agent，用户无需学习 | Agent 配置不暴露给用户 |
| N3 | **Agent 广场** — UI 让用户浏览、选择、切换 Agent | 新增页面/组件 |
| N4 | **上游兼容** — 最大程度隔离，方便跟随原作者升级 | 改动通过扩展点而非修改 |

## 三、候选方案

### 方案 A：Sidecar 进程 + 扩展模块（推荐，已选定）

```
┌──────────────────────────────────────────────────────┐
│                    ClawX (Electron)                   │
│  ┌─────────────────────────────────────────────────┐ │
│  │             原有 React 前端 (不变)               │ │
│  │  Chat | Agents | Channels | Skills | Cron | ... │ │
│  └─────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────┐ │
│  │          新增: Agent 广场页面 (扩展)              │ │
│  └─────────────────────────────────────────────────┘ │
│  ┌────────────────────┬────────────────────────────┐ │
│  │  原有 Electron     │  新增: 扩展模块             │ │
│  │  Main Process      │  (electron/extensions/)     │ │
│  │  (不变)            │  ├─ kernel-bridge.ts        │ │
│  │                    │  ├─ agent-registry.ts       │ │
│  │                    │  └─ encrypted-config.ts     │ │
│  └────────────────────┴────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
           ↕ IPC                        ↕ WebSocket
┌──────────────────────┐    ┌─────────────────────────┐
│  OpenClaw Gateway    │    │  独立内核 (Sidecar)      │
│  (原有, 不变)        │    │  Node.js 进程            │
│                      │    │  ├─ Agent Engine         │
│                      │    │  ├─ Tool Registry        │
│                      │    │  ├─ Session Manager      │
│                      │    │  └─ Agent Config (加密)   │
└──────────────────────┘    └─────────────────────────┘
```

**优点**：隔离度最高、上游兼容最好、安全性高、可扩展
**缺点**：需开发独立内核进程
**工作量**：中等偏大

### 方案 B：内嵌模块 + 编译时封装

Agent 引擎嵌入 Electron Main Process，使用 bytenode 编译为 V8 字节码。

**优点**：最简单、性能最优
**缺点**：隔离度低、上游合并风险高
**工作量**：中等

### 方案 C：Fork + Git Subtree 管理策略

使用 Git Subtree 将原 ClawX 作为子目录管理。

**优点**：代码级完全隔离
**缺点**：构建复杂、开发体验差
**工作量**：初期大

## 五、最终决策

**选定方案 A-1：Sidecar 进程 + Node.js 内核**

选择理由：
1. ChatClaw 已验证双内核架构的可行性（Go 实现），我们用 Node.js 更契合 Electron 技术栈
2. 进程隔离确保内核崩溃不影响主应用
3. 对原项目修改控制在 5 个文件以内，上游合并几乎无冲突
4. Node.js 生态中 `openclaw` SDK 已是项目依赖，可直接调用

## 六、参考项目关键借鉴点

| 来源 | 借鉴内容 |
|------|---------|
| **ChatClaw** | 双模式切换机制（chat=内核, agent=Gateway）、DeerFlow 风格 Agent 编排（Lead + Sub-Agent）、config.patch 配置同步 |
| **Clawlite** | Go Agent 进程的完整 ReAct 循环实现、WebSocket 通信协议、会话池管理、自动 Compaction、安全审批机制 |
| **OpenHarness** | AgentDefinition 模型（Pydantic）、工具白/黑名单、权限模式、Plugin 加载优先级、Hook 系统、Swarm 多 Agent 协调 |
| **OpenClaw Agent** | SOUL/IDENTITY/TOOLS/AGENTS 四文件架构、L1/L2/L3 三层提示词分离、Memory 管理机制 |

## 七、风险与应对

| 风险 | 等级 | 应对策略 |
|------|------|---------|
| 原项目 API 变更导致集成点失效 | 中 | 限制集成点到稳定 IPC 层，版本锁定 |
| 内核进程稳定性问题 | 中 | 参考现有 Gateway supervisor 机制，实现自动重启 |
| Agent 配置被逆向工程 | 低 | AES-256 加密 + V8 字节码双重保护 |
| 上游大规模重构 | 低 | 方案 A 的隔离设计使冲突最小化 |
