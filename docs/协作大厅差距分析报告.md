## 三、深度差距对比（按功能维度）

### 3.1 存储层（Store Layer）

| 能力 | 当前 | 参考 | 差距 |
|------|------|------|------|
| Hall 存储 | 基础 JSON 存储 | 同基础 | 基本一致 |
| Message 存储 | 基础 JSON 存储 | 同基础 | 基本一致 |
| TaskCard 存储 | 基础 JSON 存储 | **含预算阈值、执行锁、DOD、产物、回滚计划** | 缺少预算相关字段 |
| ExecutionLock | 类型定义存在但未使用 | 活跃/超时检测 | **完全缺失** |
| SessionKey 管理 | 字段存在但未使用 | 线程级作用域 `agent:${id}:hall:${taskId}` | **完全缺失** |
| 预算联动 | 无 | TaskCard 与 BudgetPolicy 关联 | **完全缺失** |

**结论**：存储层差距较小，但缺少预算、执行锁、会话密钥等扩展字段。

### 3.2 编排器（Orchestrator）—— **最大差距**

| 能力 | 当前 | 参考 | 差距 |
|------|------|------|------|
| 自动状态流转 | **无** | 完整状态机：discussion → assign → execution → review → completed | **完全缺失** |
| 超时处理 | **无** | 每个阶段可配置超时，超时自动流转或告警 | **完全缺失** |
| 执行锁 | **无** | 防止并发执行冲突 | **完全缺失** |
| 自动 assign | **无** | 基于角色和能力域自动选择首个执行者 | **完全缺失** |
| 自动 handoff | **无** | 检测完成信号后自动交接给下一个 | **完全缺失** |
| 自动 review | **无** | 执行完成后自动进入评审阶段 | **完全缺失** |
| 阻塞处理 | **无** | 检测到 blockers 自动暂停并通知相关人员 | **完全缺失** |
| 讨论收集 | 手动 openDiscussion | 自动收集 N 个短回复后进入 assign | **严重缺失** |
| 计划执行顺序 | 字段存在但未编排 | 自动按 plannedExecutionOrder 推进 | **严重缺失** |

**结论**：当前实现**完全没有编排器**。参考实现的 ~1,400 行 orchestrator 是协作大厅的"大脑"，负责所有自动流转逻辑。当前仅提供手动 API（assign/handoff/review），需要用户/前端手动触发每一步。

## 五、核心问题诊断

当前协作大厅只做了"界面壳子"，**缺少三个核心引擎**：

1. **编排引擎（Orchestrator）**：没有自动状态流转，所有阶段变化依赖手动 API 调用。
2. **运行时引擎（Runtime Dispatch）**：没有与 Gateway 的深度集成，Agent 回复未经清洗和结构化解析，无法驱动工作流。
3. **实时流引擎（SSE Streaming）**：没有实时推送，前端只能轮询，体验差。

### 为什么当前实现只能"展示界面"？

当前 `runtime-dispatch.ts` 的核心逻辑只有：
```typescript
// 简化为：发一条消息给 Gateway，拿到回复，存为消息
const response = await gatewayManager.rpc('chat.send', { ... });
await appendMessage({ content: response.text, ... });
```

而参考实现的运行时要做：
1. 构建复杂 prompt（含历史、角色、模式、任务上下文）
2. 调用 AgentRun（非简单 chat.send）
3. 流式读取回复，实时推送 SSE
4. 解析 `<hall-structured>` 块提取决策
5. 清洗 ANSI/工具痕迹/思考过程
6. 检测交付物类型并强制执行
7. 推断下一步动作（handoff/blocked/continue/review）
8. 更新 TaskCard 状态
9. 触发下一轮编排

## 六、建议的改进路径

### 第一阶段：核心引擎（让协作大厅"活"起来）

1. **重写 Runtime Dispatch**（最关键）
   - 适配 ClawX 的 `gatewayManager`（没有 ToolClient.agentRun，需用现有 RPC 能力模拟）
   - 实现提示词构建、结构化解析、内容清理
   - 实现交付物检测与重试机制
   - 工作量：7-10 天

2. **实现 Orchestrator**
   - 基于状态机的自动流转
   - 讨论收集 → 自动 assign → 执行监控 → 自动 handoff/review
   - 执行锁与超时处理
   - 工作量：5-7 天

3. **实现 SSE Streaming**
   - 后端 SSE endpoint
   - 草稿/终稿分离推送
   - 前端 EventSource 连接
   - 工作量：3-4 天

### 第二阶段：体验增强

4. **任务管理增强**（DOD、执行链、排程）
5. **前端交互增强**（@自动补全、实时流 UI、阶段可视化）
6. **预算治理**（阈值配置、超支拦截）

### 第三阶段：周边模块

7. **文档中心**（P1）
8. **记忆增强**（P1/P2）
9. **外部桥接**（P2）

## 七、风险评估

| 风险 | 缓解措施 |
|------|----------|
| ClawX 没有 `ToolClient.agentRun`，Runtime Dispatch 需大量适配 | 复用现有 `gatewayManager.rpc('chat.send')` + 手动构建 prompt，逐步逼近参考实现能力 |
| 编排器逻辑复杂，容易引入 bug | 先实现最小可用手动编排（API 触发），再叠加自动检测 |
| SSE 增加连接管理复杂度 | 使用现有 HostEventBus 做内部中转，前端统一 SSE 连接 |
| 上游更新冲突 | 所有新增代码严格隔离在 `electron/modules/collaboration/` 内，不修改核心文件 |

---

## 八、结论

当前协作大厅完成了：
- 数据模型定义
- 基础存储 CRUD
- 简单前端展示
- 手动 API（assign/handoff/review）

**缺失的核心能力**：
- 自动编排与状态流转
- 与 Gateway 的深度运行时集成
- 实时 SSE 流
- 交付物质量保障
- 任务生命周期管理

**要让协作大厅从"界面"变成"可用系统"，必须优先实现编排器 + 运行时派发 + SSE 流这三驾马车。**
