## 二、三种策略对比

| 维度 | A. Rebase 同步（当前） | B. 扩展点架构 | C. 深度 Fork + 选择性移植 |
|------|----------------------|--------------|------------------------|
| **同步成本** | 高（每次 3-6 小时） | 低（每次 ~30 分钟） | 按需（每次 0.5-2 小时） |
| **二开自由度** | 低（怕改太多合并难） | 中（需遵循扩展点规范） | **高（完全自由）** |
| **代码质量** | 低（硬改交织） | 高（清晰分层） | 高（完全自主） |
| **获取上游新功能** | 全量（包括不需要的） | 全量 | **选择性（只要想要的）** |
| **获取上游 bug 修复** | 自动 | 自动 | **需手动 cherry-pick** |
| **实施改造成本** | 无 | 1-2 周 | 无 |
| **长期维护风险** | 高（随二开加深爆炸） | 低 | 中（依赖上游 commit 粒度） |
| **适合二开程度** | <20% | 20%-50% | **>50%** |

## 三、深度 Fork 策略详细分析

### 3.1 优势

1. **零冲突压力** — 不再需要 rebase，不担心 upstream 更新任何文件
2. **完全自主节奏** — 何时同步、同步什么，完全由我们决定
3. **深度改造自由** — 可以重构 upstream 的核心架构（Chat、Sidebar、Store）而不受制约
4. **选择性获益** — 只 cherry-pick 有价值的 PR（如 bug 修复、新 provider），跳过不需要的功能
5. **消除 sync(main) 噪音** — 不再有那些"同步上游"的中间 commit

### 3.2 风险与挑战

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| **安全修复遗漏** | 高 | 定期（每月）审查 upstream 的 security 相关 commit |
| **cherry-pick 依赖链** | 中 | upstream 的 PR 经常依赖前序 PR，需追踪完整链路 |
| **逐步分叉增大** | 中 | 对我们修改过的文件，upstream 的同类修复需要手动理解后重新实现 |
| **API 兼容性** | 中 | upstream 更新 OpenClaw 版本时，需评估是否跟进 |
| **社区脱节** | 低 | 无法直接提交 issue/PR 给 upstream 反映二开问题 |

### 3.3 工作流程设计

```
日常开发:
  → 完全在自己的代码库工作，无任何限制

上游发布新版本时（如 v0.4.3）:
  1. 拉取 upstream 最新 tag
  2. 生成差异报告:
     git log v0.4.2-beta.2..v0.4.3 --oneline
     git diff v0.4.2-beta.2..v0.4.3 --stat
  3. 分类审查:
     ├── Bug 修复 → 评估是否影响我们，决定是否 cherry-pick
     ├── 安全修复 → 必须 cherry-pick 或手动移植
     ├── 新功能   → 评估价值，选择性移植
     └── 重构/优化 → 跳过（我们的架构可能已不同）
  4. 执行移植:
     ├── 未修改的文件 → 直接 cherry-pick（通常无冲突）
     └── 已修改的文件 → 理解 upstream 改动意图，手动实现
  5. 更新 package.json 中的 upstream 追踪字段
```

### 3.4 可行性判断

**我们的情况极度适合深度 Fork**，理由：

1. **二开程度已达 60%** — 远超 fork 阈值（50%），继续 rebase 只会越来越痛苦
2. **核心文件已大幅改造** — Chat、Sidebar、Store、Electron main 都已侵入式修改，upstream 每次更新这些文件都必冲突
3. **独有模块系统成熟** — `src/modules/` 和 `electron/modules/` 已独立运作，与 upstream 解耦度较好
4. **上游版本密集但质量参差** — 85 commits/月，不是每个都需要，选择性移植效率更高
5. **合并记录已证明痛苦** — 即使这次"顺利"也花了 3 小时，且需要恢复被删除的模块目录

## 四、推荐策略：深度 Fork + 扩展点改造混合

### 4.1 阶段划分

```
阶段 1（立即）: 宣布深度 Fork
  - 删除 upstream remote 的自动同步流程
  - 保留 upstream remote 仅用于手动 cherry-pick
  - 更新 package.json 的 upstream 追踪为纯参考

阶段 2（1-2 周内）: 扩展点改造（关键文件）
  - Sidebar → Registry 模式（消除最频繁冲突点）
  - electron/main → Lifecycle Hook（消除启动逻辑冲突）
  - src/App.tsx → 自动发现模块路由（消除路由合并冲突）
  - 这不是为了 rebase，而是为了我们自己的代码质量

阶段 3（持续）: 选择性同步流程
  - 每月检查一次 upstream 新版本
  - 只移植: 安全修复 + 高价值新功能
  - 记录移植日志到 docs/
```

### 4.2 扩展点改造优先级

| 文件 | 改造方式 | 收益 |
|------|---------|------|
| `src/components/layout/Sidebar.tsx` | Sidebar Registry | 二开导航项完全解耦 |
| `electron/main/index.ts` | Lifecycle Hook | 启动逻辑解耦 |
| `src/App.tsx` | 自动路由发现 | 路由注册解耦 |
| `src/pages/Chat/index.tsx` | Chat Extension | 聊天功能解耦 |
| 主题系统 | CSS Variable Override | 样式完全独立 |

### 4.3 选择性移植分类标准

| 类型 | 处理方式 | 优先级 |
|------|---------|--------|
| 安全漏洞修复 | **必须移植** | P0 |
| 数据损坏/丢失修复 | **必须移植** | P0 |
| 性能回归修复 | 评估影响后决定 | P1 |
| 新 Provider 支持 | 评估用户需求后决定 | P1 |
| 新 Chat 功能 | 评估与我们的改造兼容性 | P2 |
| UI/UX 改进 | 通常跳过（我们有自己的 UI） | P3 |
| OpenClaw 版本升级 | 评估兼容性后决定 | P2 |
| 代码重构 | 跳过 | P4 |

## 五、行动项

1. **立即：宣布深度 Fork** — 当前二开程度已过临界点，继续 rebase 是沉没成本
2. **保留但改造扩展点** — 不是为了 upstream 兼容，而是为了我们自己的代码质量（SRP、模块化）
3. **建立 upstream 监控机制** — star + watch upstream repo，每月审查 changelog
4. **建立移植日志模板** — 每次移植记录：源 commit、移植方式、测试结果

## 七、总结

> **核心结论：二开程度 60%，已过 fork 临界点。深度 Fork + 扩展点改造是最优策略。**

- Rebase 同步已不可持续，每次合并成本只会增加
- 深度 Fork 给予完全自主权，但需建立选择性移植纪律
- 扩展点改造不是为 upstream 兼容，而是为自身代码质量
- 投入 1-2 周改造扩展点后，日常维护成本将大幅降低
