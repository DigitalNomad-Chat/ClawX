# 二开与上游同步架构建议

> 日期: 2026-05-11
> 当前状态: 已合并 upstream v0.4.2-beta.2，工作区干净
> 核心矛盾: 深度二开需求 vs 上游同步成本

---

## 一、问题诊断: 为什么合并这么痛苦?

### 1.1 根本原因分析

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **侵入式修改** | 直接修改 `src/pages/Chat/*.tsx`、`Sidebar.tsx`、`electron/main/index.ts` | upstream 更新这些文件必冲突 |
| **功能混杂** | 二开代码散落在核心文件中，与 upstream 代码交织 | 无法区分"谁改了什么" |
| **缺少扩展点** | upstream 没有为二开预留 Hook/注册表/配置点 | 只能硬改 |
| **同步策略被动** | 等 upstream 更新后再处理冲突 | 冲突爆发式出现，难以消化 |

### 1.2 本次合并的问题映射

```
Conflict File: src/pages/Chat/index.tsx
├─ upstream: +649 行 (skill preview, beforeSend hooks, LaTeX)
├─ 我们:     +479 行 (abort state, module integration)
└─ 结果:     双方改动同一函数核心逻辑 → HIGH 冲突

Conflict File: src/components/layout/Sidebar.tsx
├─ upstream: 重构 session buckets, 添加 Dreams 导航
├─ 我们:     添加 moduleNavItems, 语义化 HTML
└─ 结果:     同一组件不同方向重构 → HIGH 冲突

Conflict File: electron/main/index.ts
├─ upstream: 添加 zoom shortcuts, extension 初始化
├─ 我们:     添加 initModules, registerExtensions
└─ 结果:     启动逻辑差异 → MEDIUM 冲突
```

### 1.3 核心结论

> **痛苦不是上游更新太频繁，而是二开方式太"侵入"。**

## 二、架构方案: 从"硬改"到"扩展"

### 2.1 目标架构

```
┌─────────────────────────────────────────────┐
│              Upstream Core (不可触碰)           │
│  ┌─────────────┐  ┌─────────────┐            │
│  │   Chat      │  │   Sidebar   │            │
│  │   (核心)     │  │   (核心)    │            │
│  └──────┬──────┘  └──────┬──────┘            │
│         │                │                    │
│    Extension Points      │                    │
│    (Hooks / Registry)    │                    │
└─────────┼────────────────┼────────────────────┘
          │                │
          ▼                ▼
┌─────────────────────────────────────────────┐
│           Our Extensions (完全隔离)            │
│  ┌─────────────┐  ┌─────────────┐            │
│  │  Collaboration│  │   Dashboard  │           │
│  │  Module      │  │   Module     │           │
│  └─────────────┘  └─────────────┘            │
│  ┌─────────────┐  ┌─────────────┐            │
│  │  Theme Plugin│  │  Layout Hook │           │
│  │  (NiceAI)    │  │  (Sidebar)   │           │
│  └─────────────┘  └─────────────┘            │
└─────────────────────────────────────────────┘
```

### 2.2 四大设计原则

| 原则 | 含义 | 实践 |
|------|------|------|
| **零侵入** | 不修改 upstream 的任何文件 | 通过扩展点注册 |
| **声明式** | 用配置代替代码修改 | JSON/YAML 配置驱动行为 |
| **可组合** | 功能以插件/模块形式独立 | 每个功能一个目录 |
| **可卸载** | 删除一个模块不影响其他 | 模块之间无硬依赖 |

## 三、具体改造建议

### 3.1 前端改造

#### 当前问题

```tsx
// src/components/layout/Sidebar.tsx (当前侵入式)
// upstream 更新了 navItems 结构，我们加了 moduleNavItems → 冲突
const navItems = [
  ...coreNavItems,
  { to: '/marketplace', icon: <Store />, label: '应用广场' },
  ...moduleNavItems.map(...), // 我们加的
];
```

#### 建议方案: Sidebar Extension Registry

```tsx
// src/extensions/sidebar-registry.ts (新增文件，upstream 无此文件)
export interface SidebarExtension {
  id: string;
  to: string;
  icon: ReactNode;
  label: string;
  order?: number;
  condition?: () => boolean; // 条件渲染
}

const sidebarExtensions: SidebarExtension[] = [];

export function registerSidebarExtension(ext: SidebarExtension) {
  sidebarExtensions.push(ext);
  sidebarExtensions.sort((a, b) => (a.order ?? 100) - (b.order ?? 100));
}

export function getSidebarExtensions(): SidebarExtension[] {
  return sidebarExtensions;
}
```

```tsx
// src/modules/collaboration/index.tsx (模块自注册)
import { registerSidebarExtension } from '@/extensions/sidebar-registry';

registerSidebarExtension({
  id: 'collaboration',
  to: '/collaboration',
  icon: <Users />,
  label: '协作大厅',
  order: 60, // 排在 Channels 之后
});
```

```tsx
// src/components/layout/Sidebar.tsx (改造一次，之后 upstream 更新也不会冲突)
// 只需在 navItems 数组最后添加一行：
const navItems = [
  ...coreNavItems.filter(...),
  ...getSidebarExtensions().map(ext => ({
    to: ext.to,
    icon: ext.icon,
    label: ext.label,
    testId: `sidebar-nav-${ext.id}`,
  })),
];
```

**收益**: 以后 upstream 更新 Sidebar.tsx，只要不删除 `getSidebarExtensions()` 这一行，就不会产生冲突。

#### Chat 页面同理

```tsx
// src/extensions/chat-registry.ts
export interface ChatExtension {
  id: string;
  beforeSend?: (message: string) => string | null; // 拦截/修改消息
  renderToolbar?: () => ReactNode; // 添加工具栏按钮
  onMessageRender?: (message: Message) => ReactNode | null; // 自定义渲染
}
```

### 3.2 后端改造

#### 当前问题

```ts
// electron/main/index.ts (当前侵入式)
// upstream 更新了启动逻辑，我们加了 initModules → 冲突
app.whenReady().then(() => {
  createWindow();
  registerIpcHandlers();
  // 我们加的:
  void initModules({ gatewayManager, clawHubService, eventBus, mainWindow });
  registerExtensions();
});
```

#### 建议方案: Lifecycle Hook System

```ts
// electron/extensions/lifecycle.ts (新增文件)
export interface AppLifecycleHooks {
  onReady?: (ctx: AppContext) => void | Promise<void>;
  onWindowCreated?: (window: BrowserWindow) => void;
  onGatewayConnected?: (gateway: GatewayManager) => void;
  onBeforeQuit?: () => void | Promise<void>;
}

const lifecycleHooks: AppLifecycleHooks[] = [];

export function registerLifecycleHooks(hooks: AppLifecycleHooks) {
  lifecycleHooks.push(hooks);
}

export async function runLifecycleHook(
  hookName: keyof AppLifecycleHooks,
  ...args: unknown[]
) {
  for (const hooks of lifecycleHooks) {
    const hook = hooks[hookName];
    if (hook) {
      try {
        await (hook as Function).apply(null, args);
      } catch (err) {
        logger.warn(`Lifecycle hook ${hookName} failed:`, err);
      }
    }
  }
}
```

```ts
// electron/modules/registry.ts (改造)
import { registerLifecycleHooks } from '../extensions/lifecycle';

export function initModules(ctx: ModuleContext) {
  for (const module of allModules) {
    if (module.registerIpc) module.registerIpc(ctx);
    if (module.lifecycleHooks) {
      registerLifecycleHooks(module.lifecycleHooks);
    }
  }
}
```

```ts
// electron/main/index.ts (改造后，只需添加 3 行)
app.whenReady().then(async () => {
  const window = createMainWindow();
  registerIpcHandlers(gatewayManager, clawHubService, window);
  
  // 新增: 运行所有模块的 onReady 钩子
  await runLifecycleHook('onReady', { gatewayManager, clawHubService, eventBus: hostEventBus, mainWindow: window });
});
```

**收益**: upstream 更新 `electron/main/index.ts` 的启动逻辑，只要不删除 `runLifecycleHook('onReady')` 这一行，就不会冲突。

### 3.3 主题系统改造

#### 当前问题

`tailwind.config.js` 和 `src/styles/globals.css` 都有冲突 — upstream 添加了设计 token，我们添加了 NiceAI 主题。

#### 建议方案: CSS Variable Override

```css
/* src/styles/theme-niceai.css (保留，upstream 无此文件) */
@layer base {
  /* 通过 CSS 变量覆盖，不修改 upstream 的 token 定义 */
  .theme-niceai {
    --background: 210 20% 97%;
    --foreground: 225 30% 8%;
    --primary: 221 100% 55%;
  }
}
```

```tsx
// src/components/theme-provider.tsx
// 通过 body class 切换主题，不修改任何 upstream 文件
export function applyTheme(theme: 'default' | 'niceai') {
  document.body.classList.remove('theme-default', 'theme-niceai');
  document.body.classList.add(`theme-${theme}`);
}
```

**收益**: 主题完全通过 CSS 变量和 class 切换实现，upstream 更新 `globals.css` 或 `tailwind.config.js` 都不会冲突。

## 四、同步策略建议

### 4.1 从"被动合并"到"主动同步"

| 策略 | 当前方式 | 建议方式 |
|------|---------|---------|
| 同步频率 | 等 upstream 发布大版本才合并 | 每周/每两周同步一次 main |
| 同步方式 | 手动 rebase，逐个解决冲突 | 自动化脚本 + 扩展点检查 |
| 冲突处理 | 爆发式处理，一次处理几十个 | 小步快跑，每次处理几个 |

### 4.2 建立同步检查清单

```bash
# scripts/sync-check.sh (可以写一个自动化脚本)
echo "=== Upstream Sync Check ==="

# 1. 检查扩展点是否被 upstream 删除
grep -r "getSidebarExtensions\|runLifecycleHook" src/ electron/ \
  || echo "WARNING: Extension points removed!"

# 2. 检查是否有新的 upstream 功能需要集成
git log upstream/main..HEAD --oneline | grep "feat\|fix" | head -20

# 3. 检查模块系统是否完整
ls src/modules/registry.ts electron/modules/registry.ts \
  || echo "ERROR: Module registry missing!"

# 4. 运行测试
pnpm test
```

### 4.3 版本追踪

`package.json` 中已有的字段要继续维护：

## 六、决策树

```
二开程度评估:
│
├─ < 20% 修改 → 保持当前方式，继续 rebase
│
├─ 20%-50% 修改 → 采用扩展点架构（本报告建议）
│   ├─ 将二开代码从核心文件提取到扩展点
│   ├─ 建立注册表/Hook 机制
│   └─ 核心文件只保留一行扩展点调用
│
└─ > 50% 修改 → 彻底 fork，独立维护
    ├─ 断开 upstream 依赖
    ├─  cherry-pick 需要的 upstream 修复
    └─  自己决定版本节奏
```

## 七、总结

> **核心建议：停止"修改核心文件"，改为"注册扩展"。**

当前痛苦的根源是二开代码和 upstream 代码在同一个文件中交织。通过建立扩展点注册机制（Sidebar Registry、Lifecycle Hooks、Chat Extensions），可以将二开代码完全隔离到独立目录，upstream 更新核心文件时几乎不会产生冲突。

**投入产出比**：
- 投入：1-2 周改造扩展点架构
- 收益：以后每次 upstream 合并从"6-8 小时"降到"30 分钟"
- 额外收益：代码更清晰，功能模块化，更容易测试和维护
